# Аудит плана и улучшения

Дата: 2026-05-09

## Что было слабым в первом плане

### 1. Слишком широкая рамка

План одновременно хотел:

- Reddit;
- X/Twitter;
- RSS;
- сайты;
- авто-постинг;
- Instagram/Facebook;
- несколько языков;
- несколько ниш;
- монетизацию.

Это слишком много для первого цикла. Такой проект легко превратить в вечную архитектуру без результата.

Правильное ограничение:

```text
1 ниша -> 3-5 типов источников -> draft queue -> ручное одобрение -> ручной/полуручной постинг
```

### 2. Не был учтён существующий проект `RedditScraper`

Уже есть рабочая база:

- `D:\Development\Parsers\RedditScraper`
- `sources.yaml`
- `scraper.py`
- `rss_scraper.py`
- `main_orchestrator.py`
- `processor_drafts.py`
- `content_vault.db`

Значит MVP не надо начинать с нуля. Надо рефакторить текущий проект в модульную систему.

### 3. Текущая БД слишком бедная

Сейчас таблица `posts` хранит:

- id
- title
- text
- upvotes
- url
- timestamp
- niche
- is_posted

Проблемы:

- нет `source_type`;
- нет `source_name`;
- нет `author`;
- нет `comments_count`;
- нет `raw_payload_json`;
- нет `content_hash`;
- нет `language`;
- нет `risk_score`;
- нет отдельного статуса для `collected`, `selected`, `drafted`, `approved`, `posted`;
- `is_posted` смешивает "использовано в драфте" и "реально опубликовано".

### 4. RSS получает искусственные 100 upvotes

Это ломает скоринг.

Нужно заменить на отдельные поля:

- `external_score`;
- `source_weight`;
- `engagement_score`;
- `quality_score`.

RSS не должен притворяться Reddit-постом.

### 5. Недостаточно provenance

Каждый драфт должен знать:

- из какого источника пришла идея;
- какие URLs использовались;
- какие факты проверены;
- какие claims рискованные;
- на каком языке сделана версия;
- на какую платформу рассчитан текст.

Без этого контент легко станет непроверяемым AI-мусором.

### 6. Нельзя строить систему вокруг browser-scraping X/Meta

X прямо запрещает non-API automation и browser scripting для автоматизации. Meta также жёстко относится к automated data collection без разрешения.

Решение:

- X использовать только через официальный API или вручную/экспортами.
- Meta не использовать как источник для scraping в MVP.
- Instagram/Facebook подключать как каналы публикации через Meta Business Suite, официальный API или проверенный scheduler.

### 7. AI-драфтер не должен создавать evergreen filler

В `processor_drafts.py` есть fallback: если хороших новостей меньше 3, генерировать evergreen topic из внутреннего знания модели.

Для контент-фермы это опасно:

- снижает связь с источниками;
- создаёт риск галлюцинаций;
- ухудшает аналитическую ценность.

Лучше:

```text
если хороших источников мало -> output: "not enough signal"
```

### 8. Нет разделения research и publishing

Нужно разделить:

- сбор источников;
- скоринг;
- editorial selection;
- генерация драфтов;
- approval queue;
- публикация;
- метрики.

Каждый шаг должен быть воспроизводимым и логируемым.

## Исправленная стратегия

Первый домен:

> AI tools / B2B SaaS

Первый технический фокус:

> расширить `RedditScraper` в `ContentFarm` без полного переписывания.

Первый бизнес-фокус:

> полезные русско- и украиноязычные дайджесты/посты по западным AI/SaaS сигналам.

## Новый MVP

```text
Reddit + RSS + Hacker News + Product Hunt + GitHub
  -> normalized source_items
  -> scoring
  -> top opportunities
  -> RU/UA drafts
  -> Obsidian approval queue
  -> manual publishing
  -> metrics
```

4. Скоринг без искусственных upvotes.
